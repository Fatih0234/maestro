# Phase 5 v2: Runtime Polish, Lifecycle Hardening, and Release Readiness

## Why this is v2
The original Phase 5 plan was written early and is now stale.

Current reality:
- Runtime completion now hands off to **human review** (`in_review`), not `done`.
- CLI is implemented with Go `flag` (not Cobra).
- Core package tests already exist and pass.
- Durable run diagnostics exist under `.contrabass/projects/<project>/runs/...`.
- There are still critical polish gaps (notably `--dry-run`, shutdown semantics, and docs drift).

This v2 plan updates Phase 5 to reflect the actual codebase and current product goals.

---

## Goal
Ship an operationally reliable CLI/runtime that is predictable in real usage:
- flags do what they say,
- shutdown is safe in both TUI/headless modes,
- retry behavior is restart-resilient,
- docs/todos match system behavior,
- release workflow is testable and reproducible.

---

## Priority Plan

## P0 — Correctness & Lifecycle Hardening (do first)

### P0.1 Wire `--dry-run` correctly
**Problem:** Flag exists but runtime does not exit after a single poll cycle.

**Tasks**
- [ ] Implement one-cycle execution path for `--dry-run` in `cmd/contrabass/main.go`.
- [ ] Ensure dry-run exits with deterministic status code.
- [ ] Ensure dry-run does not mutate issue state unexpectedly when no work is intended.

**Acceptance**
- `./contrabass --config ... --no-tui --dry-run` performs one poll cycle and exits.

---

### P0.2 Make logging flags truthful
**Problem:** `--log-level` is parsed but not effectively used.

**Tasks**
- [ ] Either wire real log-level filtering OR remove/deprecate unsupported behavior.
- [ ] Keep behavior consistent across TUI/headless modes.

**Acceptance**
- Help text and behavior match exactly.

---

### P0.3 Unify orchestrator shutdown semantics
**Problem:** `Stop()` and `shutdown()` flow can drift; cleanup guarantees must be explicit.

**Tasks**
- [ ] Ensure one authoritative shutdown path runs cleanup exactly once.
- [ ] Guarantee agent stop + runner close + recorder close + events channel close ordering.
- [ ] Prevent leaks/hangs on repeated stop signals.

**Acceptance**
- Repeated Ctrl+C / signal sequences do not deadlock or panic.

---

### P0.4 TUI and headless termination parity
**Tasks**
- [ ] Ensure `q` / `ctrl+c` in TUI always stops orchestrator before process exit.
- [ ] Ensure headless signal handler exits promptly and cleanly.
- [ ] Ensure no orphan agent processes are left behind.

**Acceptance**
- Clean exit behavior is consistent in both modes.

---

## P1 — Operational Robustness

### P1.1 Retry persistence across restart
**Problem:** Backoff intent is mostly in-memory.

**Tasks**
- [ ] Persist retry schedule (`retry_queued` + `retry_after`) through tracker/runtime state.
- [ ] Rehydrate retry-ready issues on process restart.
- [ ] Document restart behavior clearly.

**Acceptance**
- Restart does not silently lose retry intent.

---

### P1.2 Operator-visible lifecycle logs
**Tasks**
- [ ] Improve headless lifecycle logs (claim/start/retry/review-ready).
- [ ] Keep persistent diagnostics as source-of-truth.
- [ ] Ensure logs are concise but reconstructible.

**Acceptance**
- A failed/successful run can be reconstructed from logs without live TUI.

---

### P1.3 Human-review handoff UX
**Tasks**
- [ ] Add minimal review-oriented CLI utility path (list review-ready / mark done) OR explicitly defer with docs.
- [ ] Expose branch/workspace handoff info in a clear operator path.

**Acceptance**
- Human review workflow is obvious and practical from CLI/docs.

---

## Definition of Done
- [ ] P0 complete and verified
- [ ] At least one P1 robustness slice complete (retry persistence strongly preferred)
- [ ] P2 baseline (tests + CI + docs hygiene) complete
- [ ] Behavior and documentation are aligned with human-review handoff model

---

## Non-goals (for this phase)
- Multi-agent team orchestration expansion
- External tracker integrations beyond current minimal scope
- Web dashboard feature buildout
- Major architecture rewrites unrelated to runtime correctness/polish

## Implementation progress
- Added a one-shot `RunOnce()` path and wired `--dry-run` to execute a single poll cycle, then shut down cleanly.
- Reworked orchestrator shutdown so `Stop()` only requests cancellation and `shutdown()` owns cleanup exactly once.
- Added pre-start / in-loop shutdown guards to reduce new work dispatch after stop requests.
- Wired CLI log-level parsing and filtered console output by severity.
- Added regression tests for log-level parsing, shutdown cleanup, pre-cancelled runs, and one-shot shutdown.
- Verified with `go test ./...` and `go build -o /tmp/contrabass ./cmd/contrabass`.
