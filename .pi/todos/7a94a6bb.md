# Phase 2: Agent Integration — OpenCode Runner & AgentProcess

Talk to OpenCode — the most protocol-heavy part. This phase introduces HTTP client, SSE streaming, and process management.

## Goals
- Launch and manage `opencode serve` process lifecycle
- Create HTTP sessions with OpenCode server
- Submit prompts asynchronously
- Stream SSE events from OpenCode
- Handle abort and graceful shutdown

## Components

### 2.1 OpenCode Runner (`internal/agent/opencode.go`)
Manages `opencode serve` process and HTTP communication:

#### Process Management
- Spawn `opencode serve` as a subprocess (per workspace)
- Parse startup output for server address: `listening on http://127.0.0.1:9090`
- Track PID for cleanup
- Graceful stop: `kill -INT <pid>`
- Handle process exit, crashes, port conflicts

#### HTTP Client
- Base URL from parsed server address
- Optional password from config
- Keep-alive connections

#### Session Lifecycle
- `POST /session` → create session, return `{ "id": "sess-123" }`
- `POST /session/{id}/prompt_async` → submit task
  - Content-Type: `application/json`
  - Body: `{ "parts": [{ "type": "text", "text": "..." }] }`
  - Returns 204 No Content on acceptance
- `POST /session/{id}/abort` → stop current task
- `GET /event` → SSE stream for all events

#### SSE Event Handling
Key event types to parse:
- `session.status` with `status.type: idle` → session completed successfully
- `session.status` with `status.type: error` → session failed
- `server.heartbeat` → ignore (keepalive)
- `session.status` with token counts → emit token updates
- `session.output` → agent output chunks

Parse SSE format: `event: <type>\ndata: <json>\n\n`

### 2.2 AgentProcess (`internal/agent/runner.go`)
The handle the orchestrator works with:

```go
type AgentProcess struct {
    PID       int
    SessionID string
    Events    chan AgentEvent   // streams events to caller
    Done      chan error        // closed when session finishes
}
```

- `AgentEvent` struct captures: event type, token counts, output chunks, error messages
- Channels must be non-blocking from caller's perspective
- `Done` channel carries nil on success, error on failure

### 2.3 AgentEvent Types (`internal/agent/events.go`)
Typed event constants and structures:
- `EventTypeSessionStatus`
- `EventTypeTokensUpdated`
- `EventTypeOutput`
- `EventTypeError`
- `EventTypeHeartbeat`

## OpenCode Protocol Reference

```
1. Start server:
   opencode serve --port 9090
   → Parse: "listening on http://127.0.0.1:9090"

2. Create session:
   POST http://127.0.0.1:9090/session
   → { "id": "sess-123" }

3. Submit prompt:
   POST http://127.0.0.1:9090/session/sess-123/prompt_async
   Content-Type: application/json
   { "parts": [{ "type": "text", "text": "..." }] }
   → 204 No Content

4. Stream events:
   GET http://127.0.0.1:9090/event
   Accept: text/event-stream

   event: session.status
   data: {"type":"session.status","properties":{"sessionID":"sess-123","status":{"type":"idle"}}}

5. Abort (optional):
   POST http://127.0.0.1:9090/session/sess-123/abort
   → 200 OK

6. Stop server:
   kill -INT <pid>
```

### Key SSE Event Types (from `/server/#events`)

| Event | When | Action |
|-------|------|--------|
| `server.connected` | First event on connect | Ignore (connection established) |
| `server.heartbeat` | Periodic keepalive | Ignore |
| `session.status` | Session state changes | Check `properties.status.type`: `idle` = done, `error` = failed |
| `session.error` | Session error | Emit error, finish |
| `session.output` | Agent output chunks | Forward to caller |

### Session Lifecycle (from `/server/#sessions`)

```
POST /session
  → {"id": "sess-123"}

POST /session/sess-123/prompt_async
  → 204 No Content (async, no wait)

GET /event  (SSE)
  → session.status events until status.type === "idle"

POST /session/sess-123/abort  (optional)
  → 200 OK
```

## Reference: Contrabass Agent Implementation

`docs/references/contrabass/internal/agent/opencode.go` — **reference only, do not copy**

Key patterns to adapt:
- `sseReader` type: `bufio.Scanner` with 64KB buffer, handles multi-line data, event ID tracking
- `extractListeningURL()`: regex-free URL extraction from stdout lines
- `ensureServer()`: per-workspace server management with mutex map
- `startServer()`: spawn process, scan stdout for URL, timeout on startup
- `streamSessionEvents()`: SSE reader loop, filter by sessionID, non-blocking send
- `eventPayloadIdle()`: type-assertion chain for `properties.status.type`
- `eventPayloadSessionID()`: type-assertion chain for `properties.sessionID`
- Signal handling: `ExitCode() == -1` means signal-terminated (not a crash)
- Graceful stop: `SIGINT` → wait → `SIGKILL` fallback

## Open Implementation Questions

These are resolved before writing code:

1. **Workspace-per-server?** Yes — one `opencode serve` per workspace (key = workDir), matching Contrabass approach. Allows isolated sessions per issue.
2. **Port allocation:** Fixed port from config, auto-increment on collision. Default port 0 = auto-assign.
3. **Server startup timeout:** 30 seconds (matches Contrabass default).
4. **Event channel buffer:** 128 (matches Contrabass).
5. **`Close()` behavior:** Kill all managed server processes on shutdown.
